Phase 10 · Performance Measurement

Where the latency actually goes

bench/latency_bench.cpp instruments five stage boundaries end to end (network I/O → decode → demux → book update → publish) across 1,147,840 recorded deltas, and perf record (run inside the Linux Docker container from Phase 8, since perf has no real Apple Silicon macOS equivalent) independently confirms the same story from CPU sampling: the order book itself is fast. What dominates is everything around it.

Overview Flame Graph Latency Table Hot Functions Method

Three things this run proved

  • ✓ 33.5% of all CPU samples landed in nanosleep. The decoder/publisher polling loop's sleep_for(50µs) is the single largest cost in the whole profile — confirmed two independent ways (CPU sampling and the queue-wait latency numbers below), not assumed from either alone.
  • ✓ ~10% of CPU time went to measuring latency itself. The steady_clock::now() calls this instrumentation added show up as a real, non-trivial line item in the profile — an honest cost of measurement, not a free lunch.
  • ✓ 528ns mean for book.apply() itself — sub-microsecond at p50 on both platforms. The zero-allocation order book design (Phase 3) is not the bottleneck; the surrounding polling architecture is.

Captured inside the same Linux/Docker container as the flame graph above, for an apples-to-apples read against the CPU profile. tick → book update is the primary metric (network I/O received → OrderBookT::apply() returned); end to end additionally includes the publish stage. This run's inner sink is a no-op NullDeltaSink, so “publish” above measures the recording wrapper's own overhead, not real downstream I/O.

On the native macOS host, the same benchmark reports tick → book update mean 48,174ns / p50 32,417ns — roughly half the Docker container's queue-wait-dominated numbers above. book_update itself is comparable on both (528ns vs. 1,303ns mean, both sub-2µs) — the difference is the polling loop meeting Docker Desktop's coarser VM timer granularity (see docker/README.md's Phase 8 finding: sleep_for(50µs) measured ~133µs actual in-container vs. ~70µs natively), not the algorithm.

docker/ Environment & method

perf really works here, with real symbol resolution — but only by bypassing Ubuntu's version-check wrapper. /usr/bin/perf refuses to run at all ("perf not found for kernel 6.12.76"): Docker Desktop's containers share the host's LinuxKit kernel rather than running their own, so the apt-installed linux-tools-generic package (built for a generic Ubuntu kernel, 6.8.0-134) doesn't version-match. The actual binary underneath, /usr/lib/linux-tools/6.8.0-134-generic/perf, works fine when invoked directly — the perf_event_open() ABI tolerates the version skew even though the wrapper script refuses to try.
macOS-native equivalent, not run here: Instruments' Time Profiler is the closest match, but it's a GUI-driven `.trace` workflow with no clean non-interactive CLI path for this session. xcrun xctrace record --template 'Time Profiler' or the lightweight sample <pid> utility would be the CLI-adjacent options worth trying in an interactive terminal — genuinely worth doing as a cross-check on Apple Silicon specifically (this profile is arm64-on-Linux-in-a-VM; a native arm64 macOS profile would remove the virtualization variable entirely), just not something this non-interactive session could drive.
Workload: mdfh_latency_bench — 8 shards, 1,000,000 messages, 0.05% packet loss, perf record -F 999 -g (999 Hz call-graph sampling), 13,018 samples captured, 0 lost.
